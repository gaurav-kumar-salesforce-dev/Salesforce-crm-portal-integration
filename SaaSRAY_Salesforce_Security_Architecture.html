Salesforce Security Model
for SaaSRAY CRM

A practical implementation guide for roles, profiles, permission sets,
object security, field security, record sharing, and team access

Prepared as a design reference for CRM schema, backend authorization,
and admin panel development

1. What this document is for

This document explains the Salesforce security model in plain language and translates it into a design that can be used in SaaSRAY CRM. It is written so that a developer, an AI coding agent, or even a non-technical business user can understand how access should work.

The goal is simple: a user should only see the data and actions that they are truly allowed to access. That means the system must control access at the role level, profile level, permission set level, object level, field level, and record level.

2. Core idea of Salesforce security

Salesforce does not rely on one single permission check. Instead, it uses multiple layers. A user may be allowed to log in, but still not be allowed to see a specific object, a specific field, or a specific record.

• Authentication answers: Who are you?

• Role answers: Where do you sit in the business hierarchy?

• Profile answers: What is your base access?

• Permission sets answer: What extra access do you get?

• Object security answers: Can you read, create, edit, or delete this object?

• Field security answers: Can you see or update this specific field?

• Record security answers: Can you see this exact record?

A good CRM should evaluate access in the same order so that the backend is secure even if the UI is bypassed.

4. Simple example using a CRM record

Imagine an Account record named Google. John is the owner, and his manager Sarah sits above him in the role hierarchy. John's profile allows Account read and edit, but not delete. A permission set gives delete access to Opportunities only.

• John can open the Google Account because his profile allows Account read.

• John can edit the Google Account because his profile allows Account edit.

• John cannot delete the Google Account because neither profile nor permission set allows that.

• Sarah may also see John's record if role hierarchy or sharing rules allow it.

• A hidden field like Credit Score is still invisible even if John can see the Account record.

5. Roles

A role describes the user's position in the organization's hierarchy. Roles are not the same as object permissions. A role does not automatically give read or edit access to an object. Its main job is to help decide which records should be visible upward in the organization.

Example hierarchy:

• Super Admin

• Admin

• Manager

• Employee

• Read Only

In a Salesforce-style model, a manager can usually see records owned by subordinates, but subordinates do not automatically see records owned by managers.

6. Profiles

Every user must have exactly one profile. A profile is the baseline permission package for that user. This is where Salesforce-style access starts. If a profile does not allow an action, the user should not get that action unless another mechanism adds it.

• Support Profile

7. Permission Sets

Permission Sets add extra access on top of a profile. They are used when most users in a profile share common access, but a few users need a little more.

Important rule: permission sets add access, they do not remove access.

If John has a Sales Profile and receives a permission set called Opportunity Delete Access, John can now delete Opportunities even though his profile does not allow it.

8. Permission Set Groups and Muting

Salesforce also supports Permission Set Groups. A group is a container that bundles multiple permission sets into one assignable unit. This makes administration easier when a user needs several related permissions.

Example:

• Permission Set A: Export Reports

• Permission Set B: Bulk API Access

• Permission Set C: Delete Leads

• Permission Set Group: Sales Power User = A + B + C

Salesforce also supports muting inside a permission set group. Muting means the group can suppress one specific permission that is present in one of the included permission sets.

Example: a user receives a Sales Power User group, but the Delete Leads permission is muted for that group. The user gets the rest of the permissions, but not Delete Leads.

13. Role hierarchy

Role hierarchy allows access to flow upward. If a manager is above an employee, the manager can usually see the employee's records. This is very common in sales organizations.

Hierarchy example:

• CEO

• Sales Director

• Sales Manager

• Sales Rep

If a Sales Rep owns an Account, the Sales Manager above them can often see that Account, and the Sales Director can also see it.

14. Sharing rules

Sharing rules are automatic exceptions that open access for groups of users. They are used when the default OWD is too restrictive but a department or team needs broader access.

Example:

• OWD for Accounts is Private.

• All members of the Support team need read access to Accounts owned by Sales.

• A sharing rule grants Support read access to Sales-owned Accounts.

Sharing rules do not give permission to the whole object. They only share specific records or sets of records.

15. Manual sharing

Manual sharing is used for one-off cases. A record owner can explicitly share a record with another user or group.

Example:

• John owns the Google Account.

• John shares it with Mike.

• Mike can now view that specific Account record.

Manual sharing is useful when access needs are temporary or highly specific.

16. Team-based access

Team-based access allows several users to be connected to one record with different access levels. This is very useful for complex sales cycles where multiple people help manage the same customer.

17. Public groups and queues

Salesforce also uses groups and queues in record sharing and assignment. A public group is a reusable collection of users. A queue is a holding place for records that are waiting to be picked up.

Example:

• Public Group: East Sales Team

• Public Group: Finance Review Team

• Queue: Unassigned Leads

• Queue: Open Cases

Queues are especially useful for Leads and Cases. A new Lead can be assigned to a queue so that any qualified user in that queue can pick it up.

18. System Administrator

The System Administrator profile is the highest-level profile. In Salesforce, it usually has broad powers to manage users, security, configuration, and all data.

• Can manage users

• Can manage profiles and permission sets

• Can configure security settings

• Can view and modify most or all records

• Usually bypasses normal sharing restrictions

A good SaaSRAY CRM should treat super_admin similarly to Salesforce system administrators.

IMPORTANT ARCHITECTURE NOTE FOR SAASRAY CRM

SaaSRAY CRM uses a single Salesforce Integration User for all API communication with Salesforce.

Because all portal users operate through the same Salesforce account, Salesforce native ownership fields such as:

• OwnerId
• CreatedById
• LastModifiedById

cannot be used as the source of truth for portal-level ownership and security.

Example:

John creates Account A.

Sarah creates Account B.

Both records are created through the same Salesforce integration user.

Salesforce will show:

OwnerId = Integration User

CreatedById = Integration User

for both records.

As a result, Salesforce cannot distinguish which portal user actually owns each record.

To solve this, SaaSRAY CRM must maintain custom ownership fields directly on Salesforce records.

Required custom fields:

• Portal_Owner__c
• Portal_Created_By__c
• Portal_Last_Modified_By__c

Example:

Account: Google

OwnerId = Integration User

Portal_Owner__c = U001

Portal_Created_By__c = U001

Portal_Last_Modified_By__c = U002

Where:

U001 = John

U002 = Sarah

The Portal_* fields become the source of truth for:

• Record Ownership
• Record Visibility
• Sharing Rules
• Team Access
• Manager Access
• Audit Tracking

This architecture eliminates the need for a large record_ownership table in Supabase and avoids storing millions of duplicate ownership rows.

Salesforce remains the system of record for business data and ownership metadata.

Supabase remains the system of record for:

• Users
• Roles
• Profiles
• Permission Sets
• Sharing Rules
• Audit Logs
• Security Configuration

This approach is recommended for large-scale deployments containing millions of Salesforce records because it minimizes database storage requirements while preserving Salesforce-style security behavior.

20. Security flow in the backend

The backend should enforce the same rules that Salesforce enforces. The UI can hide buttons, but the API must still reject unauthorized calls.

Typical flow for a record request:

• Verify the JWT token.

• Load the current user.

• Check the user's role.

• Check profile permissions.

• Add permission-set permissions.

• Check object-level access.

• Check field-level access for the requested fields.

• Check record-level access through ownership, role hierarchy, sharing, and teams.

• Return data only if all checks pass.

24. What your CRM should implement to match Salesforce closely

If you want SaaSRAY CRM to feel like Salesforce, the most important thing is not the UI styling alone. The access model must behave like Salesforce at the backend.

• One profile per user

• Multiple permission sets per user

• Object permissions for every object

• Field permissions for sensitive columns

• Record ownership maintained through Portal_Owner__c

• Record creator tracking through Portal_Created_By__c

• Record modifier tracking through Portal_Last_Modified_By__c

• Salesforce remains the business-data store

• Supabase remains the security and authorization store

• Role hierarchy for upward visibility

• Sharing rules for cross-team access

• Manual sharing for special cases

• Team-based collaboration on records

• Strict API enforcement with 401 and 403 handling

This document can be used directly by a developer or coding agent to build database tables, API guards, admin screens, and data access logic.
